01 PL/SQL Block Structure & Control Flow

🍳 Analogy Think of a PL/SQL block like a cooking recipe: DECLARE = your ingredient list, BEGIN = the cooking steps, EXCEPTION = what to do if something burns, END = serve it.

Every PL/SQL program is built from blocks. A block has up to four sections: DECLARE (variables), BEGIN (logic), EXCEPTION (errors), and END. Only BEGIN and END are mandatory. Anonymous blocks run once; named blocks (procedures, functions) are stored and reused.

DECLARE
  v_salary  NUMBER;
  v_name    VARCHAR2(50);
BEGIN
  SELECT salary, first_name
  INTO   v_salary, v_name        -- INTO is MANDATORY
  FROM   employees
  WHERE  employee_id = 101;

  DBMS_OUTPUT.PUT_LINE(v_name || ' earns ' || v_salary);
EXCEPTION
  WHEN NO_DATA_FOUND THEN
    DBMS_OUTPUT.PUT_LINE('Employee not found.');
END;

Implicit Cursor Attributes — after any DML or SELECT INTO, Oracle populates these automatically:

  • SQL%FOUND — TRUE if the last statement affected ≥1 row
  • SQL%NOTFOUND — TRUE if the last statement affected 0 rows
  • SQL%ROWCOUNT — integer: number of rows affected

Control Structures:

-- IF
IF salary > 5000 THEN bonus := 500;
ELSIF salary > 3000 THEN bonus := 200;
ELSE bonus := 0;
END IF;

-- Basic LOOP (must use EXIT WHEN to stop)
LOOP
  counter := counter + 1;
  EXIT WHEN counter > 10;
END LOOP;

-- WHILE LOOP
WHILE counter <= 10 LOOP
  counter := counter + 1;
END LOOP;

-- FOR LOOP (automatic counter, REVERSE to go backwards)
FOR i IN 1..10 LOOP
  DBMS_OUTPUT.PUT_LINE(i);
END LOOP;
  • DDL statements (CREATE, DROP) CANNOT run in standard PL/SQL — use EXECUTE IMMEDIATE
  • DML (INSERT, UPDATE, DELETE) and SELECT INTO are fully supported
  • Nested loops use labels (<<outer>>) to EXIT a specific level

02 Cursors & Composite Data Types

👆 Analogy A cursor is like a bookmark moving through a book. An implicit cursor is automatic (self-checkout). An explicit cursor is manual — you decide when to open, read, and close it.

When a query returns multiple rows, you need an explicit cursor to process them one at a time. A %ROWTYPE record is a shortcut — one variable that holds an entire table row.

DECLARE
  CURSOR c_emp IS
    SELECT employee_id, salary FROM employees FOR UPDATE;
  v_rec c_emp%ROWTYPE;
BEGIN
  OPEN c_emp;
  LOOP
    FETCH c_emp INTO v_rec;
    EXIT WHEN c_emp%NOTFOUND;         -- stop when no more rows
    UPDATE employees
      SET salary = v_rec.salary * 1.1
      WHERE CURRENT OF c_emp;         -- update the exact fetched row
  END LOOP;
  CLOSE c_emp;
END;

The Cursor FOR Loop is the cleanest approach — it auto-opens, auto-fetches, and auto-closes:

FOR rec IN (SELECT * FROM departments) LOOP
  DBMS_OUTPUT.PUT_LINE(rec.department_name);
END LOOP;
AttributeMeaning
%ISOPENTRUE if cursor is currently open
%FOUNDTRUE if last FETCH returned a row
%NOTFOUNDTRUE if last FETCH returned nothing
%ROWCOUNTNumber of rows fetched so far
  • FOR UPDATE — locks rows so no one else can modify them
  • WHERE CURRENT OF cursor — updates/deletes the exact row just fetched
  • %ROWTYPE declares a record matching a whole table row
  • TYPE ... IS RECORD defines a custom record with chosen fields

03 Exception Handling

🧯 Analogy Exceptions are like fire extinguishers. Without one, a fire (error) burns down the building (crashes the program). With one, you catch it, handle it, and keep running.

The EXCEPTION section catches errors gracefully. It uses WHEN ... THEN handlers. WHEN OTHERS is the catch-all and must be last.

EXCEPTION
  WHEN NO_DATA_FOUND THEN
    DBMS_OUTPUT.PUT_LINE('No rows returned.');
  WHEN TOO_MANY_ROWS THEN
    DBMS_OUTPUT.PUT_LINE('Too many rows for SELECT INTO.');
  WHEN ZERO_DIVIDE THEN
    DBMS_OUTPUT.PUT_LINE('Division by zero!');
  WHEN OTHERS THEN
    DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;
TypeHow to useExample
PredefinedUse name directlyNO_DATA_FOUND, TOO_MANY_ROWS, ZERO_DIVIDE
Non-PredefinedDECLARE + PRAGMA EXCEPTION_INITORA-01400 (cannot insert NULL)
User-DefinedDECLARE + RAISECustom business rules
 
-- Non-predefined: link a name to an Oracle error number
DECLARE
  e_null_insert EXCEPTION;
  PRAGMA EXCEPTION_INIT(e_null_insert, -01400);
BEGIN
  INSERT INTO employees(employee_id) VALUES(NULL);
EXCEPTION
  WHEN e_null_insert THEN DBMS_OUTPUT.PUT_LINE('Cannot be NULL!');
END;

-- User-defined: raise your own error
DECLARE
  e_too_young EXCEPTION;
BEGIN
  IF age < 18 THEN RAISE e_too_young; END IF;
EXCEPTION
  WHEN e_too_young THEN DBMS_OUTPUT.PUT_LINE('Must be 18+');
END;

-- RAISE_APPLICATION_ERROR: range -20000 to -20999
RAISE_APPLICATION_ERROR(-20001, 'Custom error message here');
  • SQLCODE — returns the error number (negative for Oracle errors)
  • SQLERRM — returns the error message text
  • PRAGMA EXCEPTION_INIT is for non-predefined errors, NOT predefined ones
  • Valid range for RAISE_APPLICATION_ERROR: -20000 to -20999

04 Procedures & Functions

🔧 Analogy A procedure is a power drill — it does work. A function is a measuring tape — it does work AND gives you back a number. You can hang a measuring tape inside a blueprint (SQL), but you can't hang a power drill there.
FeatureProcedureFunction
Returns valueNo (uses OUT params)Yes (RETURN)
Use in SQL SELECT❌ No✅ Yes (if no DML)
InvocationEXECUTE or BEGIN...ENDexpression context
Header keywordIS / AS (not DECLARE)IS / AS + RETURN type
 
-- Procedure
CREATE OR REPLACE PROCEDURE raise_salary(
  p_id  IN  employees.employee_id%TYPE,
  p_pct IN  NUMBER,
  p_new OUT NUMBER
) IS
BEGIN
  UPDATE employees SET salary = salary * (1 + p_pct/100)
  WHERE employee_id = p_id;
  SELECT salary INTO p_new FROM employees WHERE employee_id = p_id;
END raise_salary;

-- Function (safe for SQL — no DML inside)
CREATE OR REPLACE FUNCTION get_tax(p_salary NUMBER) RETURN NUMBER IS
BEGIN
  RETURN p_salary * 0.12;
END get_tax;

-- Usage in SQL:
SELECT first_name, get_tax(salary) FROM employees;
  • Parameters: IN (read-only input), OUT (write-only output), IN OUT (both)
  • Functions with DML (INSERT/UPDATE/DELETE) inside cannot be called in SQL context
  • Local subprograms: procedures/functions defined in another's DECLARE section — scoped locally
  • Benefits of subprograms: Modularity, Reusability, Security, Performance

05 Triggers

⏰ Analogy A trigger is like a motion-sensor alarm. It sleeps until something happens (someone enters the room / a row is updated). Then it automatically fires and runs its code.

Triggers fire automatically when a DML event occurs on a table or view. You don't call them — Oracle calls them for you.

CREATE OR REPLACE TRIGGER audit_salary
BEFORE UPDATE OF salary ON employees
FOR EACH ROW                          -- row-level: fires once per row
BEGIN
  IF :NEW.salary < :OLD.salary THEN  -- :OLD = before, :NEW = after
    RAISE_APPLICATION_ERROR(-20001, 'Salary cannot decrease!');
  END IF;
  INSERT INTO salary_audit
    VALUES(:OLD.employee_id, :OLD.salary, :NEW.salary, SYSDATE);
END;
FeatureRow-Level (FOR EACH ROW)Statement-Level
FiresOnce per affected rowOnce per DML command
:OLD / :NEW✅ Available❌ Not available
Use caseAudit each row changeLog the operation itself
  • BEFORE — fires before DML executes (can prevent the operation)
  • AFTER — fires after DML executes
  • INSTEAD OF — replaces DML on a view
  • Mutating Table Error — a row trigger cannot SELECT from the table it is modifying
  • Triggers cannot receive parameters — they are event-driven
  • :NEW/:OLD work only in row-level triggers
  • Manage: ALTER TRIGGER name DISABLE/ENABLE; DROP TRIGGER name;

06 Data Warehousing

🏛 Analogy A database is a busy post office processing today's mail. A Data Warehouse is the national archive — it stores every letter ever sent, organized for historical research, not for processing new mail.

A Data Warehouse (DW) is subject-oriented, integrated, time-variant, and non-volatile. It's designed for analysis (OLAP), not day-to-day transactions (OLTP). Data gets in through the ETL process: Extract from sources → Transform (clean, standardize) → Load into the warehouse.

ComponentDescription
EDW (Enterprise DW)Centralized, enterprise-wide decision support
Data MartDepartment-specific subset of the DW
ODS (Operational DS)Near real-time operational data, not historical

Schema Types:

SchemaStructureDimensions
Star1 Fact table + multiple Dimension tablesDenormalized (flat)
SnowflakeStar with normalized dimensionsNormalized (multiple tables)
GalaxyMultiple Fact tables sharing dimensionsMixed
  • Fact Table — contains measurable data (sales amount, quantity)
  • Dimension Table — provides context (who, what, when, where)
  • Star Schema dimensions are denormalized (not normalized)
  • DW is for OLAP (analysis), NOT OLTP (transactions)

07 Data Mining & CRISP-DM

⛏ Analogy Data Mining is like panning for gold in a river. The river (data warehouse) has tons of water (raw data) — you pan through it systematically to find gold nuggets (useful patterns).

Data Mining is the extraction of useful patterns and knowledge from large datasets. It's the analysis step that builds on top of a Data Warehouse.

CRISP-DM PhaseWhat happens
1. Business UnderstandingDefine the problem and goals
2. Data UnderstandingExplore and collect data
3. Data PreparationClean, transform, select features
4. ModelingApply algorithms (Decision Tree, K-means…)
5. EvaluationAssess model quality vs. business goals
6. DeploymentPut the model into production
TypeAlso calledTechniques
PredictiveSupervised LearningClassification, Regression
DescriptiveUnsupervised LearningClustering, Association
  • KDD (Knowledge Discovery in Databases) has 9 steps including pre-processing, transformation, algorithm selection
  • CRISP-DM has 6 phases and is the industry standard methodology
  • Data Warehouse = where data lives; Data Mining = how you extract knowledge from it

08 Supervised Learning

🏫 Analogy Supervised learning is like a teacher grading homework. You show the model examples with correct answers (labeled data), and it learns the pattern to predict future answers.

Supervised learning uses labeled data — each training example has a known output. The goal is to learn a mapping from input features to output labels.

AlgorithmKey ConceptFormula/Idea
Decision Tree (ID3)Entropy + Information GainIG = Entropy(S) − Entropy(Attribute)
Naïve BayesProbability + IndependenceP(A|B) = P(B|A)×P(A) / P(B)
Neural Network (ANN)Input → Hidden → Output layersWeights + Bias + Activation
 
-- Decision Tree: ID3 Key Concepts
Entropy(S) = -Σ p_i × log2(p_i)
-- Perfect split: Entropy = 0
-- Maximum uncertainty: Entropy = 1

Information Gain = Entropy(S) - Weighted_Entropy(Attribute)
-- Higher IG → better attribute to split on

-- Example: Entropy(S) = 0.94, Entropy(Outlook) = 0.50
-- IG = 0.94 - 0.50 = 0.44  ✓
  • Root Node — topmost decision node (highest IG attribute)
  • Leaf Node — terminal node with class label (no more splits)
  • Splitting — dividing data based on an attribute value
  • Pruning — removing sub-nodes to reduce overfitting (opposite of splitting)
  • Naïve Bayes assumes all features are independent of each other
  • Neural Networks: Input Layer → Hidden Layer(s) → Output Layer

09 Unsupervised Learning

🔍 Analogy Unsupervised learning is like sorting a pile of foreign coins without knowing the currencies. You group them by size, color, and weight — no labels, just pattern recognition.

Unsupervised learning finds hidden structure in unlabeled data. There is no correct answer to train against — the algorithm discovers groups or rules on its own.

AlgorithmTypeKey Concepts
K-meansClusteringk centroids, iterative assignment until convergence
AprioriAssociation RulesSupport, Confidence, Downward Closure
 
-- K-means Steps:
1. Choose k (number of clusters)
2. Initialize k centroids randomly
3. Assign each point to nearest centroid
4. Recalculate centroids (mean of each cluster)
5. Repeat steps 3-4 until centroids stop changing

-- Stopping condition: centroids no longer move (convergence)

-- Apriori Key Measures:
Support({A,B})    = transactions with A∩B / total transactions
Confidence(A→B)  = transactions with A∩B / transactions with A

-- Downward Closure Property:
-- If {A,B} is frequent → {A} and {B} must also be frequent
-- This allows Apriori to prune infrequent itemsets early
  • K-means is unsupervised — it has no labels, only coordinates
  • Support = how often itemset appears in all transactions
  • Confidence = P(B | A) — probability of B given A was purchased
  • Apriori's Downward Closure: all subsets of a frequent itemset must be frequent
  • K-means stops when centroids stop changing (convergence), not after fixed iterations
